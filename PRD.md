# TimeTrain（时间小火车）产品需求说明书 v1.0 开发版
## 一、核心锁定决策（开发不可修改）
以下为产品核心规则，开发过程中不得变更：
1.  **核心隐喻锁定**：车厢=仅30分钟时间容器，乘客=独立任务；车厢不可拖拽，仅支持增删操作；仅乘客（任务）支持拖拽交互
2.  **时间与火车规则**：火车随系统时间流逝自动行驶，当前时间对应的时间线标识直接展示在对应车厢上，无页面固定时间线
3.  **任务状态锁定**：仅支持`pending(未反馈)`/`done(已完成)`/`failed(未完成)`三态，仅通过车厢底部按钮修改状态，无任何自动标记规则（包括已过时间段自动标记）
4.  **同步策略锁定**：本地操作实时写入localStorage，距离上次操作30秒空闲后自动同步至Vercel KV
5.  **任务排程锁定**：任务（乘客）需按分类管理；拖拽乘客至车厢间隙为插入操作，插入位置前的车厢/时间保持不变，后续车厢内已有的乘客整体顺延30分钟；拖拽乘客至已有空车厢为任务分配操作
6.  **容错机制锁定**：全局提供撤销按钮，支持最多回溯最近10步操作
7.  **统计规则锁定**：仅当当日所有带任务的车厢均标记`done`/`failed`状态后，计算并展示当日完成率；当日所有带任务车厢均标记为`done`时，触发全屏庆祝动效
8.  **日期管理锁定**：页面打开默认加载当日数据，支持日期选择器切换任意历史日期，每日数据独立持久化存储，切换日期时完整加载当日计划与状态
9.  **统计视图锁定**：提供月历视图专属入口，月历上展示每日完成率，点击对应日期可跳转至当日火车计划页面
10. **布局规则锁定**：全设备（PC/平板/手机）统一采用**上半屏火车轨道区，下半屏任务分类区**的响应式布局，无左右分栏结构
11. **功能边界锁定**：当前版本不实现成就系统、积分系统、AI推荐、家长端、语音提醒、多孩子账户切换功能

## 二、核心概念映射
| 产品概念 | 对应现实含义 | 开发约束规则 |
|----------|--------------|--------------|
| 火车     | 单日完整时间计划 | 由多节车厢按时间顺序线性排列组成，随系统时间自动向前行驶 |
| 车厢     | 30分钟固定时长的时间容器 | 1. 不可拖拽、不可调换顺序，仅支持新增/删除操作；2. 每节车厢仅可容纳1名乘客（1个任务）；3. 车厢顺序与时间顺序完全绑定，不可变更 |
| 乘客     | 一项独立的待办任务 | 1. 页面内唯一可拖拽的交互元素；2. 按分类归属管理；3. 可分配至空车厢，或拖拽插入生成新车厢 |
| 时间线标识 | 当前系统时间 | 1. 展示在当前时间对应的车厢上，无页面固定时间线；2. 随系统时间每分钟刷新，自动匹配对应车厢 |
| pending状态 | 任务未反馈 | 车厢初始默认状态，无任务的空车厢固定为此状态 |
| done状态 | 任务已完成 | 点击车厢「✔」按钮触发，仅手动可修改 |
| failed状态 | 任务未完成 | 点击车厢「✖」按钮触发，仅手动可修改 |

## 三、时间系统
### 3.1 时间单位
系统固定最小时间单位：**30分钟=1节车厢**，所有时间标签对齐整点(:00)或半点(:30)，不支持其他分钟数。

### 3.2 火车行驶规则
- 火车整体随系统时间流逝，自动向左匀速行驶，行驶速度与真实时间流速完全匹配
- 当前系统时间对应的车厢，高亮展示时间线标识：车厢内左侧垂直彩色实线，顶部标注当前实时时间（HH:MM），每分钟刷新
- 用户可手动左右拖拽火车整体，查看过去/未来的车厢，松手后2秒自动吸附回当前时间对应的车厢居中位置

### 3.3 时间范围规则
- 单日时间范围限制为`00:00 ~ 23:30`，不可超出自然日边界
- 初始默认生成当日`17:00~22:00`的空白车厢，共10节
- 支持在火车头部、尾部、车厢间隙新增车厢，自动生成对应时间段

### 3.4 日期管理规则
- 页面首次加载，默认获取系统当前日期，加载当日的计划数据
- 顶部提供日期选择器，支持切换至任意历史/未来日期，切换时完整加载对应日期的车厢、任务、状态数据
- 每日数据独立存储，互不影响，切换日期时不修改其他日期的数据
- 历史数据永久保留，无自动清理规则

## 四、车厢系统（核心规则：不可拖拽，仅可增减）
### 4.1 车厢核心规则
- 每节车厢唯一对应一个30分钟的连续时间段，时间标签生成后固定不变，仅删除车厢时移除，如果车厢删除时其中有乘客，则乘客返回任务库
- 每节车厢最多容纳1名乘客（1个任务），空车厢无任务关联
- 车厢不可拖拽、不可调换顺序，仅支持新增、删除两种编辑操作
- 车厢状态仅由用户手动点击按钮修改，无任何自动变更逻辑

### 4.2 车厢数据结构
```typescript
export interface Carriage {
  id: string; // 全局唯一UUID
  date: string; // 所属日期，格式YYYY-MM-DD
  startTime: string; // 时间段起始，格式HH:MM
  endTime: string; // 时间段结束，格式HH:MM
  taskId: string | null; // 关联任务ID，空车厢为null
  status: "pending" | "done" | "failed"; // 任务状态，默认pending
  order: number; // 当日火车内的排序序号，从1开始递增，与时间顺序完全绑定
  createdAt: number; // 创建时间戳
  updatedAt: number; // 最后修改时间戳
}
```

### 4.3 车厢新增操作（仅支持3种方式）
1.  **首尾扩展新增**：点击火车最左侧/最右侧的「+」按钮，在头部/尾部新增一节空白车厢，自动生成对应时间段（头部向前推30分钟，尾部向后推30分钟）
2.  **间隙插入新增**：点击两节车厢之间的「+」按钮，在当前位置插入一节空白车厢，插入位置前的车厢时间保持不变，后续所有车厢的`startTime`/`endTime`整体顺延30分钟，`order`序号同步更新
3.  **任务拖拽插入新增**：拖拽乘客（任务）至两节车厢之间的间隙，松手后在该位置插入一节新车厢，自动分配该任务至车厢，后续车厢时间整体顺延30分钟

### 4.4 车厢删除操作
1.  **点击删除**：点击车厢右上角的「×」按钮，删除当前车厢；若车厢已有任务，弹出二次确认弹窗，确认后删除车厢，任务自动返回任务库
2.  **批量删除**：长按车厢进入多选模式，勾选多节车厢后，点击批量删除按钮，一次性删除选中车厢

### 4.5 车厢删除约束
- 删除车厢后，后续所有车厢的`startTime`/`endTime`整体向前顺延30分钟，`order`序号同步更新
- 删除车厢不会删除关联的任务，仅解除任务与车厢的绑定，任务返回任务库
- 当日至少保留1节车厢，不可全部删除

## 五、任务（乘客）系统
### 5.1 任务分类规则
任务分类默认为4类，不可新增/删除分类，所有任务必须归属其中一个分类，任务库按分类分区展示：
- 学习类
- 生活类
- 活动类
- 其他类

|------|----------|-----------|--------|
| 学习类 | 数学 | 📐 | #FFB84D |
| 学习类 | 语文 | 📖 | #4DA6FF |
| 学习类 | 英语 | 🔤 | #FF7A45 |
| 学习类 | 阅读 | 📚 | #52C41A |
| 生活类 | 吃饭 | 🍚 | #FFC53D |
| 生活类 | 洗漱 | 🧼 | #36CFC9 |
| 生活类 | 睡觉 | 😴 | #85A5FF |
| 活动类 | 运动 | 🏃 | #FF4D4F |

| 活动类 | 游戏 | 🎮 | #722ED1 |
| 其他类 | 其他 | 📝 | #d4d4d4 |

- 任务支持新增、编辑、删除（因为每天的作业和活动可能会不同）

### 5.3 任务数据结构
```typescript
export interface Task {
  id: string; // 全局唯一UUID
  name: string; // 任务名称，最长10个字符
  icon: string; // Emoji图标
  color: string; // 十六进制颜色码
  category: "study" | "life" | "activity" | "other"; // 所属分类
  isPreset: boolean; // 是否为系统预置任务
  isHidden: boolean; // 是否在任务库中隐藏，仅对预置任务生效
  createdAt: number; // 创建时间戳
  updatedAt: number; // 最后修改时间戳
}
```

### 5.4 任务管理操作
1.  **新增任务**：点击任务库对应分类的「+新增任务」按钮，弹出表单，填写名称、选择Emoji、颜色、分类，确认后添加至对应分类的任务库
2.  **编辑任务**：点击任务卡片的编辑图标，修改名称、Emoji、颜色、分类，修改后同步更新所有已分配该任务的车厢展示
3.  **隐藏/删除任务**：预置任务仅支持隐藏，点击隐藏后从任务库收起，不影响历史数据；自定义任务支持删除，删除后解除所有车厢与该任务的绑定，对应车厢变为空车厢

### 5.5 任务拖拽排程核心规则
1.  **分配至空车厢**：拖拽任务库中的乘客（任务）至空白车厢，松手后任务自动分配至该车厢，车厢`taskId`绑定该任务ID，任务库中该任务变为半透明状态（标识已分配）
2.  **插入新车厢**：拖拽任务至两节车厢之间的间隙，松手后在该位置插入一节新车厢，自动绑定该任务，后续车厢时间整体顺延30分钟
3.  **替换已有任务**：拖拽任务至已分配任务的车厢，松手后替换原任务，原任务自动返回任务库，新任务绑定至该车厢
4.  **取消分配**：拖拽车厢内的任务至任务库区域，松手后解除任务与车厢的绑定，车厢变为空车厢，任务恢复至任务库正常状态

## 六、执行反馈与统计系统
### 6.1 状态修改规则
- 每个已分配任务的车厢底部，固定展示「✔ 完成」「✖ 未完成」两个按钮，空车厢不展示
- 点击「✔」：车厢状态变更为`done`，视觉变为绿色渐变背景，任务图标右上角添加绿色✓角标
- 点击「✖」：车厢状态变更为`failed`，视觉变为浅红灰色背景，任务图标右上角添加红色✕角标
- 再次点击当前已激活的按钮，状态恢复为`pending`，视觉恢复默认样式
- 仅点击按钮可修改状态，无任何自动修改逻辑

### 6.2 撤销功能
- 页面顶部固定展示「撤销」按钮，仅在有可回溯操作时可点击
- 记录用户最近10步可撤销操作，包括：车厢增删、任务分配/替换/取消、状态修改、任务编辑
- 点击撤销按钮，回溯至上一步操作，恢复操作前的所有数据状态
- 切换日期后，清空当前日期的撤销操作记录，不可跨日期撤销

### 6.3 完成率计算规则
- 触发条件：当日所有已分配任务的车厢，均已标记为`done`或`failed`状态（无pending状态的任务车厢）
- 计算公式：`完成率 = 标记为done的任务车厢数 ÷ 当日总任务车厢数（不含空车厢）× 100%`
- 完成率实时展示在页面顶部，仅满足触发条件时展示具体数值，未满足时展示「待完成全部任务标记」

### 6.4 庆祝动效规则
- 触发条件：当日所有任务车厢均标记为`done`状态（完成率100%）
- 动效内容：全屏彩色纸屑庆祝动效，不阻塞页面操作
- 触发限制：只要全部状态标记为`done`，即可触发庆祝动效，重复标记为100%完成时也重复触发

## 七、月历视图系统
### 7.1 入口与展示规则
- 页面顶部日期选择器旁，固定展示「月历统计」按钮，点击进入月历视图页面
- 按月展示自然月日历，每日单元格内展示：日期、当日完成率（有数据时）
- 无任务数据的日期，仅展示日期；完成率100%的日期单元格为绿色渐变；≥60%为蓝色渐变；＜60%为灰色背景

### 7.2 交互规则
- 支持左右滑动切换上一个/下一个自然月
- 点击有数据的日期单元格，自动跳转回火车主页面，并加载该日期的计划数据
- 点击「返回今日」按钮，快速跳转回当前日期的火车主页面

## 九、数据持久化与同步策略
### 9.1 存储架构
采用`localStorage实时本地存储 + Vercel KV云端定期同步`的双层架构，本地优先，云端备份。

### 9.2 本地存储规则
- 存储技术：浏览器localStorage
- 写入时机：用户每一次操作后立即写入，延迟＜5ms，确保刷新不丢失
- 存储内容：用户UUID、任务库全量数据、每日计划车厢数据、撤销操作记录、用户偏好设置

### 9.3 云端同步规则
- 存储技术：Vercel KV（Redis）
- 同步触发时机：距离用户上次操作完成30秒空闲后自动触发全量同步；切换日期时，立即同步当前日期的最终数据
- 同步逻辑：
  1. 页面加载时，先读取localStorage数据立即渲染，同时异步请求Vercel KV数据
  2. 若KV数据的最后修改时间戳晚于本地数据，以KV数据为准覆盖本地，并刷新页面渲染
  3. 冲突处理：以`lastModified`时间戳为唯一判断标准，取最新版本数据

### 9.4 离线与用户识别规则
- 网络断开时，所有操作正常写入localStorage，不触发云端同步；网络恢复后，自动触发一次全量数据同步
- 无账号登录系统，首次访问页面时自动生成UUID作为用户唯一标识，存入localStorage永久使用；所有云端数据以UUID为唯一Key进行隔离，不收集任何个人身份信息

## 十一、API接口规范
### 通用响应格式
所有接口响应均为JSON格式，结构统一：
```json
{
  "code": 200 | 400 | 500,
  "message": "success | 错误信息",
  "data": {} | null
}
```

### 接口详情
1.  **任务库接口**
    - 接口地址：`/api/tasks`
    - 请求方法：GET
    - 请求参数：`userId`（string，必填）
    - 响应数据：`data`为Task[]全量任务数组
    ---
    - 接口地址：`/api/tasks`
    - 请求方法：POST
    - 请求体：
    ```json
    {
      "userId": "string",
      "tasks": "Task[]"
    }
    ```
    - 响应数据：`data`为更新后的Task[]数组

2.  **每日计划接口**
    - 接口地址：`/api/plan`
    - 请求方法：GET
    - 请求参数：`userId`（string，必填）、`date`（string，YYYY-MM-DD，必填）
    - 响应数据：`data`为当日DailyPlan对象
    ---
    - 接口地址：`/api/plan`
    - 请求方法：POST
    - 请求体：
    ```json
    {
      "userId": "string",
      "date": "string",
      "plan": "DailyPlan"
    }
    ```
    - 响应数据：`data`为更新后的DailyPlan对象

3.  **批量同步接口**
    - 接口地址：`/api/sync`
    - 请求方法：POST
    - 请求体：
    ```json
    {
      "userId": "string",
      "tasks": "Task[]",
      "plans": "DailyPlan[]",
      "lastModified": "number"
    }
    ```
    - 响应数据：`data`为云端最新全量数据，用于冲突覆盖
